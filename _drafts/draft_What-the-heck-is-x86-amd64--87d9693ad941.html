What the heck is x86_amd64?

This a short post that can hopefully clear up this question for anyone who’s done fresh linux installs and wondered what that term means. 

Well it all starts out with some of the first x86 processors.

One of the amazing and terrible things about the x86 architecture is that it’s maintained backwards compatibility since 1976. This has been a competitive advantage, but the boot process is a pile of hacks. Each time a new iteration comes out, a new step gets added to the process. That’s right, when your fancy new computer starts up, it thinks it’s an 8086.

The first mode (called ‘real mode’) is a 16 bit mode that original x86 chips use. The second is ‘protected mode’. This 32 bit mode adds new things on top of real mode and the final mode is called ‘long mode’, and it’s 64 bits.

You see friends, when the industry was undergoing the transition from 32 to 64 bits, there were two options: the first was Intel’s Itanium 64-bit architecture. It did away with all of the stuff I just told you about. But that meant that programs had to be completely recompiled from scratch for the new chips. 

Intel’s big competitor at the time, AMD, saw an opportunity here, and released a new set of chips called amd64. These chips were backwards compatible, and so you could run both 32 and 64 bit programs on them. Itanium wasn’t doing well enough on the market to make the pain worth it, so Intel released new chips that were compatible with amd64. The resulting architecture was then called x86_64, the one we’re using today. The moral of the story? Intel was the savior we didn’t deserve; and we rejected them.
